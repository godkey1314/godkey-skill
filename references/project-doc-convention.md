# 项目文档管理规范

本文档定义了跨项目的文档创建和维护规则。文档的目的是**留痕**，不是凑数。

---

## 一、文档目录结构

多模块项目（有子仓库/子模块）在根目录创建 `xxxx_doc/`，单项目在项目目录下创建 `doc/`：

```
project/
├── src/                    # 代码
├── doc/                    # 或 xxxx_doc/（多模块项目）
│   ├── 01-产品介绍/
│   ├── 02-使用指南/
│   ├── 03-系统架构/
│   ├── 04-功能设计/
│   ├── 05-数据库/
│   ├── 06-开发规范/
│   ├── 07-部署运维/
│   ├── 08-资源文件/
│   ├── 09-AI开发日志/
│   └── 10-开发计划/
└── CLAUDE.md
```

使用此 skill 内置的模板初始化：`assets/project-template/`。项目启动时，将此模板复制到项目的 `doc/` 或 `xxxx_doc/` 目录下。

## 二、各目录的创建与更新时机

| 目录 | 何时创建 | 何时更新 |
|------|---------|---------|
| **01-产品介绍** | 项目启动时 | 产品定位/核心特性变更时 |
| **02-使用指南** | 功能基本稳定后 | 用户操作流程变更时 |
| **03-系统架构** | 项目启动时 | 架构调整/技术选型变更时 |
| **04-功能设计** | 新功能开发完成后同步 | 功能上线后与实际代码对齐（不是开发中边改边写） |
| **05-数据库** | 首次建表时 | 每次数据库变更时（加表、改字段、迁移） |
| **06-开发规范** | 项目启动时 | 规范变更时 |
| **07-部署运维** | 首次部署前 | 环境/配置/流程变更时 |
| **08-资源文件** | 按需 | 按需 |
| **09-AI开发日志** | **每次对话必须记录** | 每次对话（只要产生代码变动） |
| **10-开发计划** | 新功能/重构/大修复前 | 方案确定后即完成，一般不反复修改 |

## 三、开发计划（10-开发计划）

### 3.1 何时创建

以下场景**必须**先创建开发计划再动手：

- 新功能开发（涉及文件 > 3 个）
- 重构（改动范围跨模块）
- 架构调整
- 数据质量/安全等专项修复方案
- 用户明确要求"先出方案"

### 3.2 何时不创建

- 修一个小 bug（单文件、局部改动）
- 文案调整
- 配置修改
- 格式修正

### 3.3 目录命名

```
10-开发计划/
├── YYYY-MM-DD/              # 当天只有一个计划
│   └── 方案名称.md
├── YYYY-MM-DD_01/           # 当天多个计划 → 日期_序号
├── YYYY-MM-DD_02/
└── README.md
```

## 四、AI 开发日志（09-AI开发日志）

### 4.1 何时记录

每次对话只要产生**代码文件落盘变动**，就必须记录日志。

不记日志的情况：
- 纯问答、只读排查
- 讨论方案但无代码改动

### 4.2 日志命名

```
09-AI开发日志/日志/
└── YYYYMMDD_SessionNNN_任务名.md
```

Session 编号接续已有日志。同一连续任务不重复开新日志。

---

## 五、功能设计（04-功能设计）

### 5.1 何时更新

**功能开发完成、代码合并后**，对照实际代码同步功能设计文档。

不在开发过程中频繁更新——开发过程中的变更记录在开发日志中。

### 5.2 更新内容

- API 路径、入参、出参
- 数据库表结构
- 业务流程图
- 关键设计决策

---

## 六、不能做的

### 6.1 禁止无脑建文档

以下行为一律禁止：

- 改一行代码就建一个开发计划
- 在功能设计里记录开发过程中的临时变更（应记在日志）
- 同一个功能建多个重复文档

### 6.2 禁止文档与代码断层

- 已上线的功能，功能设计文档必须与代码一致
- 已变更的数据表，数据库文档必须同步更新
- 不允许代码已经改成 A 了，文档还写着 B

### 6.3 禁止只写不改

文档不是写到一半就扔在那的：
- 功能设计文档以最终上线代码为准
- 如果开发计划中的方案在执行过程中有实质变更，在日志中记录偏离原因，不反复修改计划文档
