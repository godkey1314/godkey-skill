# 阅读纪律

本条纪律解决这个问题：**AI 看了少量文件就声称"已全面了解"，结果改代码时遗漏关键依赖和约束。**

---

## 核心原则

**在你输出"已理解"之前，必须能回答以下三个问题：**

1. 我读了哪些文件？每个文件的角色是什么？
2. 这些文件之间的依赖关系是怎样的？（谁调谁）
3. 有没有还没读但可能受影响的文件？

如果任一问题答不上来 → 还没读完，继续读。

---

## 阶段一：读规范（先于读代码）

在打开任何代码文件之前，必须先搜索和阅读项目的规范文档：

### 必须搜索的文件类型

```
**/*规范*    **/*标准*    **/*guide*
**/*convention*    **/*style*    **/CLAUDE.md
**/README.md    **/架构*    **/ARCHITECTURE*
**/开发指南*    **/CONTRIBUTING*
```

### 提取内容

从规范中提取与当前任务相关的约束，至少包括：

- 代码组织规则（文件命名、目录结构、模块边界）
- 分层约束（每层能做什么、不能做什么）
- 编码风格（命名、格式、注释规则）
- 禁止事项（明确列出"不要做"的事）

如果项目没有规范文档，以 `code-principles.md` 中的通用原则作为判断标准。

### 附加一：识别项目使用的框架

在开始读代码前，必须先确认项目用了哪些框架/库及其版本：

1. 检查依赖声明文件（`requirements.txt` / `package.json` / `go.mod` / `Cargo.toml` 等）
2. 列出所有关键框架的名称和版本
3. 确认这些框架在磁盘上的实际位置（虚拟环境、node_modules、GOPATH 等）

### 附加二：阅读框架源码和约束

对于代码中实际使用的每个框架/库的功能，必须找到框架源码并阅读：

1. **找到位置**：确定框架代码在磁盘上的路径（不是假设，不是猜测）
2. **阅读相关源码**：打开框架中对应模块的源文件，理解它的实际行为
3. **查明约束**：框架对并发安全、会话管理、连接池、生命周期有什么约束？
4. **对照使用**：项目代码是否遵守了这些约束？

这一步不是"如果涉及"——**只要项目用了框架，就必须走这一步。**

禁止行为：
- 禁止凭训练数据中对同名框架的记忆猜测其行为（版本不同行为可能不同）
- 禁止只看框架的公开 API 名称而不看实现细节
- 禁止假设框架是安全的——框架的默认行为往往是最大的坑

## 阶段二：读代码（追到底）

### 步骤 1：确定入口

找到与任务直接相关的入口文件。

入口文件 = 用户提到的文件 / 接口对应的 Controller / 报错堆栈的最上层 / 需求描述中的核心模块。

### 步骤 2：向下追踪

从入口文件开始，沿着调用链逐层向下。**如果涉及框架基类或内部工具库，必须打开源码阅读：**

```
入口文件
  → 它调用了谁？
    → 被调用方又在哪一层？
      → 被调用方又调用了谁？
        → 追到底（数据层实现、外部 API 调用、框架源码）
```

**追到底意味着打开框架源码**，而不是看到基类方法名就猜测行为。

### 步骤 3：向上追溯继承链

如果要修改的类继承自某个基类或实现了接口，必须先完整阅读：

```
要改的子类
  → 它的父类/基类提供了什么？
    → 基类定义了哪些抽象方法/钩子？
      → 基类的调用约定和限制是什么？
        → 追到顶（最顶层基类/接口定义）
```

向上追溯的目的是确保：理解基类的设计意图和约定，不在子类中违反基类契约，正确使用基类提供的钩子和工具方法。

### 步骤 4：向上追溯调用方

找到所有可能受改动影响的调用方：

```
要改的文件
  → 谁引用了它？
    → 调用方在哪里？
      → 调用方被谁调？
        → 追到顶（接口入口、定时任务入口）
```

### 步骤 5：横向对比

检查是否存在类似的代码模式：

- 同一目录下有没有结构类似的文件？
- 有没有其他 Mapper/Service/Controller 和当前文件做类似的事？
- 如果有，都读一遍，确认逻辑一致性

### 步骤 6：输出文件清单

在动手写代码之前，必须输出以下清单：

```markdown
## 涉及文件清单

### 直接改动
- `path/file_a.py` — 角色：数据表 A 的 Mapper，需要新增 xx 查询方法
- `path/file_b.py` — 角色：xx 的 Service，需要编排新的查询逻辑

### 间接相关（调用方）
- `path/file_c.py` — 角色：定时任务，调用了 file_b 的方法

### 间接相关（被调用方/基类）
- `path/file_d.py` — 角色：BaseTask 基类，file_c 继承自它

### 已排除
- `path/file_e.py` — 虽然同目录，但处理不同业务对象，不涉及
```

这个清单就是你的"阅读证明"。没有清单 = 没读完。

## 阶段三：审查/重构时的附加要求

代码审查和重构是 AI 最容易出错的阶段。额外要求：

### 审查时：逐条检查清单（必须逐项输出结果）

以下 15 条不可笼统说"通过了"，每条必须输出检查结果。

#### 1. 分层检查
打开每个变更文件，确认它属于哪一层。对照 [layer-thinking.md](layer-thinking.md) 判断是否越界。
**输出**：逐文件标注"该层规范-检查结果"。

#### 2. SQL 归属检查
搜索所有变更文件中的 `text(` / `f"SELECT` / `f"INSERT` / `f"UPDATE` / `f"DELETE` / raw SQL 字符串。
**输出**：列出每个 SQL 出现位置，标注是否在数据层。

#### 3. 借道访问检查
检查每个 Service 的 `__init__` 或构造函数，确认注入的依赖是否属于自己模块。
**输出**：列出所有跨模块注入，标注是否合理。

#### 4. 代码重复检查
列出所有同类文件（所有 Mapper、所有 Service、所有 Task）。逐对比它们的 public 方法签名和实现逻辑。
**输出**：重复清单（"文件 A 的 X 方法 = 文件 B 的 Y 方法"），重复次数 ≥ 3 → P2，≥ 5 → P1。

#### 5. 热路径检查
找到 Dashboard、首页、高频轮询等接口。检查它们是否每次都实时计算。
**输出**：每个高频接口标注"实时/快照/缓存"，实时且数据量大 → 标记。

#### 6. 并发安全检查
搜索 `asyncio.gather` / `Promise.all` / `goroutine` / `threading`。对每个并发点：
- 共享了什么资源（数据库会话、连接、文件）？
- 该资源是否文档声明为并发安全？
- 并发数量是否有上限？

**不要假设任何库是并发安全的。** 对使用的每个异步资源，必须检查其文档或源码中的并发约束。
**输出**：每个并发点标注"资源类型-是否安全-是否有上限"。

#### 7. 值语义检查
搜索所有涉及数值/布尔/空值的赋值和比较：`if value` / `if not value` / `value or default` / `value if value else None` / `data.get("key")`。
**输出**：列出每个 falsy 判断，标注是否可能把合法 0/空字符串/空列表当成 None。

#### 8. 类型完整性检查
搜索所有类型注解（`Optional`/`List`/`Dict`/`Union` 等），对照文件顶部的 import。
**输出**：列出所有使用但未导入的类型。

#### 9. 通用组件检查
检查 TaskExecutor、框架中间件、公共工具类中是否包含特定业务领域的判断逻辑。
**输出**：列出每个业务语义侵入点。

#### 10. 前端状态检查
检查前端文件的渲染函数/computed/watch。计算属性是否在每次调用时重建对象？
**输出**：列出每个重建对象的位置，标注是否会导致状态丢失。

#### 11. 前端异步检查
检查前端文件中的 `async` 函数和 API 调用。快速连续调用时旧请求是否会覆盖新请求？
**输出**：列出每个无竞态保护的异步操作。

#### 12. 测试覆盖检查
列出所有测试文件。逐个判断是纯函数测试还是集成测试。对新增的 Mapper/Service/Task，是否有对应的集成测试？
**输出**：缺失集成测试的模块清单。

#### 13. 魔法值检查
搜索代码中的裸数字和裸字符串（`== 0` / `== "active"` / `status = "pending"`），判断是否有对应枚举。
**输出**：列出所有魔法值及应使用的枚举。

#### 14. 废弃代码检查
搜索 `getattr` / `hasattr` / `TODO` / `FIXME` / `已移除` / `待删除` / `临时` / `兼容` / `deprecated`。
**输出**：列出每个疑似兼容/废弃代码的位置。

#### 15. 文档同步检查
对照 [project-doc-convention.md](references/project-doc-convention.md)，本次改动需要哪些文档动作？
**输出**：文档动作清单（日志/计划/功能设计/架构）。

### 重构时

1. **先理解旧架构**：为什么当初这样设计？改了会影响谁？
2. **列出所有下游依赖**：全部找完才能动手
3. **重构 = 先清理再新增**：废弃代码 → 删；旧逻辑 → 搬，而不是兼容
4. **不分步"慢慢改"**：同一个重构任务中，不留下"先这样后面再改"的过渡态

---

## 禁止行为

- 禁止看入口文件 + 几个直接引用就声称"已全面了解"
- 禁止跳过基类/父类直接看子类实现
- 禁止基于文件名猜测文件角色而不打开阅读
- 禁止在审查时说"看起来没问题"而不逐条对照规范
- 禁止跳过横向对比（发现重复代码而不指出）
