# 阅读纪律

本条纪律解决这个问题：**AI 看了少量文件就声称"已全面了解"，结果改代码时遗漏关键依赖和约束。**

---

## 核心原则

**在你输出"已理解"之前，必须能回答以下三个问题：**

1. 我读了哪些文件？每个文件的角色是什么？
2. 这些文件之间的依赖关系是怎样的？（谁调谁）
3. 有没有还没读但可能受影响的文件？

如果任一问题答不上来 → 还没读完，继续读。

---

## 阶段一：读规范（先于读代码）

在打开任何代码文件之前，必须先搜索和阅读项目的规范文档：

### 必须搜索的文件类型

```
**/*规范*    **/*标准*    **/*guide*
**/*convention*    **/*style*    **/CLAUDE.md
**/README.md    **/架构*    **/ARCHITECTURE*
**/开发指南*    **/CONTRIBUTING*
```

### 提取内容

从规范中提取与当前任务相关的约束，至少包括：

- 代码组织规则（文件命名、目录结构、模块边界）
- 分层约束（每层能做什么、不能做什么）
- 编码风格（命名、格式、注释规则）
- 禁止事项（明确列出"不要做"的事）

如果项目没有规范文档，以 `code-principles.md` 中的通用原则作为判断标准。

### 附加一：识别项目使用的框架

在开始读代码前，必须先确认项目用了哪些框架/库及其版本：

1. 检查依赖声明文件（`requirements.txt` / `package.json` / `go.mod` / `Cargo.toml` 等）
2. 列出所有关键框架的名称和版本
3. 确认这些框架在磁盘上的实际位置（虚拟环境、node_modules、GOPATH 等）

### 附加二：阅读框架源码和约束

对于代码中实际使用的每个框架/库的功能，必须找到框架源码并阅读：

1. **找到位置**：确定框架代码在磁盘上的路径（不是假设，不是猜测）
2. **阅读相关源码**：打开框架中对应模块的源文件，理解它的实际行为
3. **查明约束**：框架对并发安全、会话管理、连接池、生命周期有什么约束？
4. **对照使用**：项目代码是否遵守了这些约束？

这一步不是"如果涉及"——**只要项目用了框架，就必须走这一步。**

禁止行为：
- 禁止凭训练数据中对同名框架的记忆猜测其行为（版本不同行为可能不同）
- 禁止只看框架的公开 API 名称而不看实现细节
- 禁止假设框架是安全的——框架的默认行为往往是最大的坑

## 阶段二：读代码（追到底）

### 步骤 1：确定入口

找到与任务直接相关的入口文件。

入口文件 = 用户提到的文件 / 接口对应的 Controller / 报错堆栈的最上层 / 需求描述中的核心模块。

### 步骤 2：向下追踪

从入口文件开始，沿着调用链逐层向下。**如果涉及框架基类或内部工具库，必须打开源码阅读：**

```
入口文件
  → 它调用了谁？
    → 被调用方又在哪一层？
      → 被调用方又调用了谁？
        → 追到底（数据层实现、外部 API 调用、框架源码）
```

**追到底意味着打开框架源码**，而不是看到基类方法名就猜测行为。

### 步骤 3：向上追溯继承链

如果要修改的类继承自某个基类或实现了接口，必须先完整阅读：

```
要改的子类
  → 它的父类/基类提供了什么？
    → 基类定义了哪些抽象方法/钩子？
      → 基类的调用约定和限制是什么？
        → 追到顶（最顶层基类/接口定义）
```

向上追溯的目的是确保：理解基类的设计意图和约定，不在子类中违反基类契约，正确使用基类提供的钩子和工具方法。

### 步骤 4：向上追溯调用方

找到所有可能受改动影响的调用方：

```
要改的文件
  → 谁引用了它？
    → 调用方在哪里？
      → 调用方被谁调？
        → 追到顶（接口入口、定时任务入口）
```

### 步骤 5：横向对比

检查是否存在类似的代码模式：

- 同一目录下有没有结构类似的文件？
- 有没有其他 Mapper/Service/Controller 和当前文件做类似的事？
- 如果有，都读一遍，确认逻辑一致性

### 步骤 6：输出文件清单

在动手写代码之前，必须输出以下清单：

```markdown
## 涉及文件清单

### 直接改动
- `path/file_a.py` — 角色：数据表 A 的 Mapper，需要新增 xx 查询方法
- `path/file_b.py` — 角色：xx 的 Service，需要编排新的查询逻辑

### 间接相关（调用方）
- `path/file_c.py` — 角色：定时任务，调用了 file_b 的方法

### 间接相关（被调用方/基类）
- `path/file_d.py` — 角色：BaseTask 基类，file_c 继承自它

### 已排除
- `path/file_e.py` — 虽然同目录，但处理不同业务对象，不涉及
```

这个清单就是你的"阅读证明"。没有清单 = 没读完。

## 阶段三：审查/重构时的附加要求

代码审查和重构是 AI 最容易出错的阶段。额外要求：

### 审查时：逐项检查清单（必须逐项输出结果）

以下检查项不可笼统说"通过了"，每条必须输出具体检查结果。检查的是**模式**而非特定语法——不同语言实现方式不同，但违规模式是通用的。

#### 1. 分层越界
每个变更文件属于哪一层？有没有越界行为？
- 业务层中是否出现了数据查询语句的直接拼接？
- 数据层中是否出现了业务条件判断？
- 接口层中是否出现了数据库操作？
**输出**：逐文件标注层级归属及越界项。

#### 2. 依赖借道
每个模块的依赖注入是否合理？有没有通过不相关的模块间接获取资源（如借用其他模块的数据库连接/会话）？
**输出**：列出所有跨模块资源借用的位置。

#### 3. 代码重复
列出所有同类文件（如同目录下的所有数据访问组件、所有业务服务、所有任务）。它们的公共方法签名和实现逻辑是否雷同？
**输出**：重复清单，标注重复次数。≥ 3 处标记，≥ 5 处提级。

#### 4. 热路径
高频访问的接口是否每次都实时计算？有没有利用缓存、快照或预计算结果？
**输出**：每个高频接口标注计算模式，实时扫描且数据量大 → 标记。

#### 5. 并发安全
代码中是否存在同时发起多个异步/并发操作的位置？这些操作共享了什么资源（数据库会话、网络连接、文件句柄）？
- 共享资源是否声明为并发安全？
- 并发数量是否有上限控制？
**不要假设任何库是并发安全的。** 必须检查所用框架/库的文档或源码中的并发约束。
**输出**：每个并发点标注共享资源及其并发安全性。

#### 6. 值语义（零值 vs 空值）
代码中对数值/布尔/空值的判断是否区分了"值为零"和"值不存在"？有没有用真假值判断（truthy/falsy）代替了空值检查？
**输出**：列出每个可疑的真假值判断位置。

#### 7. 类型与声明一致性
所有类型声明（如 Optional、nullable、泛型）是否完整导入？方法声明的返回类型与实际返回值是否一致？
**输出**：列出类型声明缺失或不一致的位置。

#### 8. 通用组件污染
框架执行器、中间件、公共工具类中是否混入了特定业务领域的判断逻辑？
**输出**：列出所有业务语义侵入通用组件的位置。

#### 9. 前端状态与渲染
前端渲染逻辑中，派生数据是否正确使用了缓存/计算属性机制？是否存在每次渲染都重建对象导致状态丢失的问题？
**输出**：列出每个可疑的位置。

#### 10. 前端异步竞态
前端的异步请求是否有竞态保护？快速连续操作时旧请求的响应是否会覆盖新请求？
**输出**：列出每个无保护的非幂等异步操作。

#### 11. 测试覆盖
测试是否只覆盖了纯函数，而没有验证数据库查询、服务编排、任务执行的完整链路？新增的组件是否有对应的集成测试？
**输出**：列出缺失集成测试的模块。

#### 12. 魔法值
代码中是否存在有业务含义的裸常量（数字、字符串）而未使用枚举或命名常量？
**输出**：列出所有魔法值及应替代的常量/枚举。

#### 13. 兼容垃圾
代码中是否存在为已废弃字段/方法保留的兼容逻辑、兜底值、fallback？是否存在标记"待删除""临时方案"但未清理的代码？
**输出**：列出每个疑似位置。

#### 14. 文档同步
对照 [project-doc-convention.md](references/project-doc-convention.md)，本次改动需要哪些文档动作？
**输出**：列出需要的文档动作。

### 重构时

1. **先理解旧架构**：为什么当初这样设计？改了会影响谁？
2. **列出所有下游依赖**：全部找完才能动手
3. **重构 = 先清理再新增**：废弃代码 → 删；旧逻辑 → 搬，而不是兼容
4. **不分步"慢慢改"**：同一个重构任务中，不留下"先这样后面再改"的过渡态

---

## 禁止行为

- 禁止看入口文件 + 几个直接引用就声称"已全面了解"
- 禁止跳过基类/父类直接看子类实现
- 禁止基于文件名猜测文件角色而不打开阅读
- 禁止在审查时说"看起来没问题"而不逐条对照规范
- 禁止跳过横向对比（发现重复代码而不指出）
