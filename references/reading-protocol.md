# 阅读纪律

本条纪律解决这个问题：**AI 看了少量文件就声称"已全面了解"，结果改代码时遗漏关键依赖和约束。**

---

## 核心原则

**在你输出"已理解"之前，必须能回答以下三个问题：**

1. 我读了哪些文件？每个文件的角色是什么？
2. 这些文件之间的依赖关系是怎样的？（谁调谁）
3. 有没有还没读但可能受影响的文件？

如果任一问题答不上来 → 还没读完，继续读。

---

## 阶段一：读规范（先于读代码）

在打开任何代码文件之前，必须先搜索和阅读项目的规范文档：

### 必须搜索的文件类型

```
**/*规范*    **/*标准*    **/*guide*
**/*convention*    **/*style*    **/CLAUDE.md
**/README.md    **/架构*    **/ARCHITECTURE*
**/开发指南*    **/CONTRIBUTING*
```

### 提取内容

从规范中提取与当前任务相关的约束，至少包括：

- 代码组织规则（文件命名、目录结构、模块边界）
- 分层约束（每层能做什么、不能做什么）
- 编码风格（命名、格式、注释规则）
- 禁止事项（明确列出"不要做"的事）

如果项目没有规范文档，以 `code-principles.md` 中的通用原则作为判断标准。

### 附加一：识别项目使用的框架

在开始读代码前，必须先确认项目用了哪些框架/库及其版本：

1. 检查依赖声明文件（`requirements.txt` / `package.json` / `go.mod` / `Cargo.toml` 等）
2. 列出所有关键框架的名称和版本
3. 确认这些框架在磁盘上的实际位置（虚拟环境、node_modules、GOPATH 等）

### 附加二：阅读框架源码和约束

对于代码中实际使用的每个框架/库的功能，必须找到框架源码并阅读：

1. **找到位置**：确定框架代码在磁盘上的路径（不是假设，不是猜测）
2. **阅读相关源码**：打开框架中对应模块的源文件，理解它的实际行为
3. **查明约束**：框架对并发安全、会话管理、连接池、生命周期有什么约束？
4. **对照使用**：项目代码是否遵守了这些约束？

这一步不是"如果涉及"——**只要项目用了框架，就必须走这一步。**

禁止行为：
- 禁止凭训练数据中对同名框架的记忆猜测其行为（版本不同行为可能不同）
- 禁止只看框架的公开 API 名称而不看实现细节
- 禁止假设框架是安全的——框架的默认行为往往是最大的坑

## 阶段二：读代码（追到底）

### 步骤 1：确定入口

找到与任务直接相关的入口文件。

入口文件 = 用户提到的文件 / 接口对应的 Controller / 报错堆栈的最上层 / 需求描述中的核心模块。

### 步骤 2：向下追踪

从入口文件开始，沿着调用链逐层向下。**如果涉及框架基类或内部工具库，必须打开源码阅读：**

```
入口文件
  → 它调用了谁？
    → 被调用方又在哪一层？
      → 被调用方又调用了谁？
        → 追到底（数据层实现、外部 API 调用、框架源码）
```

**追到底意味着打开框架源码**，而不是看到基类方法名就猜测行为。

### 步骤 3：向上追溯继承链

如果要修改的类继承自某个基类或实现了接口，必须先完整阅读：

```
要改的子类
  → 它的父类/基类提供了什么？
    → 基类定义了哪些抽象方法/钩子？
      → 基类的调用约定和限制是什么？
        → 追到顶（最顶层基类/接口定义）
```

向上追溯的目的是确保：理解基类的设计意图和约定，不在子类中违反基类契约，正确使用基类提供的钩子和工具方法。

### 步骤 4：向上追溯调用方

找到所有可能受改动影响的调用方：

```
要改的文件
  → 谁引用了它？
    → 调用方在哪里？
      → 调用方被谁调？
        → 追到顶（接口入口、定时任务入口）
```

### 步骤 5：横向对比

检查是否存在类似的代码模式：

- 同一目录下有没有结构类似的文件？
- 有没有其他 Mapper/Service/Controller 和当前文件做类似的事？
- 如果有，都读一遍，确认逻辑一致性

### 步骤 6：输出文件清单

在动手写代码之前，必须输出以下清单：

```markdown
## 涉及文件清单

### 直接改动
- `path/file_a.py` — 角色：数据表 A 的 Mapper，需要新增 xx 查询方法
- `path/file_b.py` — 角色：xx 的 Service，需要编排新的查询逻辑

### 间接相关（调用方）
- `path/file_c.py` — 角色：定时任务，调用了 file_b 的方法

### 间接相关（被调用方/基类）
- `path/file_d.py` — 角色：BaseTask 基类，file_c 继承自它

### 已排除
- `path/file_e.py` — 虽然同目录，但处理不同业务对象，不涉及
```

这个清单就是你的"阅读证明"。没有清单 = 没读完。

## 阶段三：审查/重构时的附加要求

代码审查和重构是 AI 最容易出错的阶段。额外要求：

### 审查时

1. **读规范在先**：先重温项目规范，建立当前判断标准
2. **读变更清单**：确认所有改动文件的边界
3. **逐文件、逐层检查**：每个文件对照 [layer-thinking.md](layer-thinking.md) 检查层级归属
4. **检查依赖链完整性**：是否因为新代码而引入了不合理的依赖方向
5. **横向对比查重复**：同类文件（如多个 Mapper、多个 Service）是否有结构相同仅参数不同的方法？3 个以上重复 → 标记，15 个以上 → P1 问题
6. **检查热路径**：高频访问的接口是否每次都实时计算？是否有缓存或快照机制？
7. **检查值语义**：0、空字符串、None 是否被正确区分？有没有 falsy 检查替代 None 检查？
8. **检查通用组件**：通用基础设施是否被业务语义污染？
9. **检查前端通用规范**：状态管理（computed vs 函数重建）、异步竞态、请求防重、格式化复用、样式原则（参考 code-principles.md §十一）
10. **检查异步/并发安全**：异步会话是否被多协程共享、并发是否有上限、共享状态是否有锁（参考 code-principles.md §十二）
11. **检查框架约束**：对代码中使用的每个框架/库，检查其文档或源码中关于并发安全、连接池、会话管理的约束和已知陷阱。不要假设框架是安全的——框架的默认行为往往是最大的坑
12. **检查值语义**：`0`/`""`/`[]` 和 `None`/`null` 是否区分、falsy 检查是否替代了 None 检查（参考 code-principles.md §八 + §12.2）
13. **检查类型完整性**：所有类型注解的导入是否完整、返回值类型是否匹配（参考 code-principles.md §12.3）
14. **输出**：问题清单，每个问题关联具体文件和规范条款

### 重构时

1. **先理解旧架构**：为什么当初这样设计？改了会影响谁？
2. **列出所有下游依赖**：全部找完才能动手
3. **重构 = 先清理再新增**：废弃代码 → 删；旧逻辑 → 搬，而不是兼容
4. **不分步"慢慢改"**：同一个重构任务中，不留下"先这样后面再改"的过渡态

---

## 禁止行为

- 禁止看入口文件 + 几个直接引用就声称"已全面了解"
- 禁止跳过基类/父类直接看子类实现
- 禁止基于文件名猜测文件角色而不打开阅读
- 禁止在审查时说"看起来没问题"而不逐条对照规范
- 禁止跳过横向对比（发现重复代码而不指出）
