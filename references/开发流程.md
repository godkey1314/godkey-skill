# 开发流程

适用场景：新功能开发、Bug 修复、代码重构、模块添加。

---

## 第一阶段：建立判断标准

**先读规范，再读代码。**

1. 搜索并阅读项目的规范文档（代码规范、架构说明、CLAUDE.md 等）
2. 提取与当前任务相关的约束：代码组织规则、分层约束、编码风格、禁止事项
3. **观察项目文档体系**：列出已有文档目录，对照 `assets/project-template/` 发现缺失
4. 如果文档体系不存在或不完整 → 用模板初始化
5. **输出**：当前任务涉及哪些规范约束？

如果项目没有规范文档，以模板 `06-开发规范/代码规范.md` 中的通用原则作为判断标准。

---

## 第二阶段：识别框架

1. 检查依赖声明文件（requirements.txt / package.json / go.mod / Cargo.toml 等），列出关键框架名称和版本
2. 找到框架在磁盘上的实际位置（虚拟环境、node_modules、GOPATH 等）
3. 打开框架源码，查明关键约束：并发安全、会话管理、连接池、生命周期
4. **禁止凭训练数据猜测框架行为**——版本不同行为可能不同

---

## 第三阶段：理解代码关系（关卡 1：读全）

### 向下追踪

```
入口文件 → 调用链逐层向下 → 追到底（框架源码）
```

### 向上追溯继承链

```
要改的子类 → 父类/基类提供什么？ → 基类的约定和限制？ → 追到顶
```

### 向上追溯调用方

```
要改的文件 → 谁引用它？ → 谁调调用方？ → 追到顶（接口入口/定时任务入口）
```

### 横向对比

检查同目录下是否有结构类似的文件，确认逻辑一致性。

### 输出文件清单（必须）

```markdown
## 涉及文件清单

### 直接改动
- `path/file_a.ext` — 角色：xx，需要 xx 改动

### 间接相关（调用方）
- `path/file_c.ext` — 角色：xx，调用了 file_a

### 间接相关（被调用方/基类）
- `path/file_d.ext` — 角色：BaseXxx 基类

### 已排除
- `path/file_e.ext` — 虽然同目录，但处理不同业务对象，不涉及
```

没有清单 = 没读完。

---

## 第四阶段：范式与定层（关卡 2：定层）

### 范式优先

写任何代码前，先找到项目中已有类似实现作为参考：

1. 找到与当前任务功能相似的已有模块
2. 阅读其文件组织、命名、方法签名、错误处理方式
3. 新代码与它们保持一致
4. 如果没有类似范式 → 以简单为先

### 层级决策树

```
这段代码做的事是？
├─ 定义 URL 路径、HTTP 方法、参数来源？
│   → A 层（接口/接入层）
│
├─ 判断"能不能做""要不要做""先后顺序"？
│   → B 层（业务/编排层）
│
├─ 拼 SQL、操作数据库、定义 ORM 模型？
│   → C 层（数据/持久层）
│
├─ 纯数学计算、统计、数据转换，不涉及业务判断？
│   → D 层（独立计算层）
│
├─ 对接外部 API，做数据格式转换？
│   → D 层（独立能力层，如 Provider/Gateway）
│
├─ AI 推理、分析、生成？
│   → D 层（独立能力层，如 Agent/Skill）
│
├─ 横切关注点（日志、缓存、权限）？
│   → 独立工具/中间件/装饰器
│
└─ 说不清属于哪一层？
    → 停下来，重新理解架构再决定
```

### 跨层污染模式

| 污染模式 | 表现 | 修正 |
|---------|------|------|
| **数据操作上移** | 业务层中出现数据查询语句拼接 | 下沉到数据层 |
| **业务下移** | 数据层中出现业务条件判断 | 上移到业务层 |
| **接口膨胀** | 接口层中写大量业务处理逻辑 | 下沉到业务层 |
| **借道访问** | 业务层通过不相关模块获取底层资源 | 使用自己模块的依赖 |
| **跨层跳跃** | 接口层直接调用数据层，跳过业务层 | 中间插入业务层 |
| **计算混入** | 在业务层或数据层中写大量计算/统计 | 抽取到独立计算层 |

## 第五阶段：执行与迭代精炼（关卡 3-4）

### 清理纪律（关卡 3）

- 废弃字段/方法/导入 → 直接删除
- 不留兼容桥、墓碑注释、fallback 兜底
- 不留"以防万一"的代码

### 自检循环（关卡 4）

1. 严格按改动清单执行，不改清单外的文件
2. 对照项目规范逐条自检，每项输出结果
3. 违规 → 修复 → 重新检查
4. 循环直到零违规，最多 3 轮

---

## 重构专项要求

1. **先理解旧架构**：为什么当初这样设计？改了会影响谁？
2. **列出所有下游依赖**：全部找完才能动手
3. **重构 = 先清理再新增**：废弃代码删，旧逻辑搬，不兼容
4. **不分步"慢慢改"**：同一个重构任务不留过渡态

---

## 禁止行为

- 禁止看入口文件 + 几个直接引用就声称"已全面了解"
- 禁止跳过基类/父类直接看子类实现
- 禁止基于文件名猜测文件角色而不打开阅读
- 禁止凭训练数据经验写代码，必须先找项目范式
- 禁止未确认框架版本和源码就开始写代码
- 禁止修改子类而不读父类/基类/接口
- 禁止"顺手"做用户未要求的功能或重构
- 禁止保留废弃代码、墓碑注释、"以防万一"的兼容代码
- 禁止创建无业务价值的抽象层
