# 开发流程

适用场景：新功能开发、Bug 修复、代码重构、模块添加。

---

## 全局锁：开发流程必须从头到尾顺序执行，不得跳过任何 Pass。

---

## Pass 0：建立判断标准（关卡 0：读规范）

**目标**：先懂规则，再定方案。在接触任何代码之前，先建立什么是好代码的判断标准。

### 读规范

1. **确认 `06-开发规范/` 存在且有内容**（前置关卡已检查，此处二次确认）
2. 通读 `06-开发规范/` 中的所有规范文档
3. **通读 `03-系统架构/`**：了解分层设计、技术选型、模块依赖关系
4. 提取与当前任务相关的约束条款，列成清单
5. 如果项目没有自己的规范文档，以模板 `assets/project-template/06-开发规范/代码规范.md` 中的通用原则作为基准

### 识别技术栈

1. 检查依赖声明文件，列出关键框架/库的名称和版本
2. **确认框架信息**：
   - 源码可访问 → 打开源码，查明关键约束
   - 远程包管理器 → 查阅对应版本的官方文档
   - 二进制/闭源 → 查阅公开文档
3. 重点查明：并发安全、会话管理、连接池、生命周期约束
4. **禁止凭训练数据猜测框架行为**——版本不同行为可能不同
5. 如果框架信息无法获取，阻塞并询问用户

**输出**：规范约束清单 + 框架约束清单（可逐条对照的具体条款，不是笼统说"注意分层"）

---

## Pass 1：设计方案与开发计划（关卡 1：设计方案）

**目标**：先画图纸，再施工。代码改动不是从读代码开始，是从写方案开始。

### 进入条件

1. Pass 0 已通过（已建立判断标准）
2. **先读 `04-功能设计/`**：找到与本次任务相关的功能设计文档，了解蓝图要求
3. **先读 `09-AI开发日志/`**：读最近 2-3 篇日志，了解项目当前状态和历史上下文

### 强制规则

1. **在进入 Pass 2 之前**，必须先完成本 Pass 并输出开发计划文档
2. 计划文档落盘后，才能继续——**没有方案文档 = 不许读代码，更不许写代码**
3. 小改动（单文件且 <20 行）可跳过 Pass 1，直接进入 Pass 2

### 方案内容要求

开发计划文档必须包含以下章节：

```markdown
## 背景与目标
- 为什么要做这个改动
- 要达到什么效果

## 涉及范围
- 涉及模块/文件清单
- 影响的上游调用方
- 影响的下游被依赖方

## 实现方案
- 核心思路（用哪个模块/哪个函数，怎么改）
- 分层设计（改动落在哪几层，每层做什么）
- 关键决策与取舍

## 影响分析
- 对现有功能的影响
- 对数据/数据库的影响（如有）
- 对接口/API 的影响（如有）

## 测试要点
- 核心路径验证
- 边界条件
- 异常路径
```

### 文档落盘

- **存放位置**：`10-开发计划/YYYY-MM-DD/方案名称.md`
- **怎么放**：使用 Write 工具，写完后向用户确认位置
- **落盘后**：告诉用户"方案已输出到 `10-开发计划/xxxx`，开始进入代码阶段"

### 绝对禁止

- **禁止把方案内容写到 `09-AI开发日志/`**——开发日志是 post-code 操作记录，开发计划是 pre-code 设计文档。两者的区别：
  - 计划：代码还没写，先设计 "准备怎么改"
  - 日志：代码已经改了，记录 "实际改了啥"
- 禁止流水账式方案——"在 xx 文件加个函数然后调用"不叫方案，叫笔记

## Pass 2：理解代码关系（关卡 2：读全）

**目标**：完整理解代码，能画出依赖关系。

### 向下追踪
入口文件 → 调用链逐层向下 → 追到底（框架源码）

### 向上追溯继承链
要改的子类 → 父类/基类提供什么？ → 基类的约定和限制？ → 追到顶

### 向上追溯调用方
要改的文件 → 谁引用它？ → 谁调调用方？ → 追到顶（接口入口/定时任务入口）

### 横向对比
检查同目录下是否有结构类似的文件，确认逻辑一致性。找到项目已有范式参考。

**输出**：完整文件清单（必须，没有清单 = 没读完）

```markdown
### 直接改动
- `path/file_a.ext` — 角色：xx，需要 xx 改动

### 间接相关（调用方）
- `path/file_c.ext` — 角色：xx，调用了 file_a

### 间接相关（被调用方/基类）
- `path/file_d.ext` — 角色：BaseXxx 基类

### 已排除
- `path/file_e.ext` — 同目录但处理不同业务对象，不涉及
```

## Pass 3：范式与定层（关卡 3：定层）

**目标**：每行新代码都知道放哪里、怎么写。

**对照 `03-系统架构/`**：确认项目实际分层和模块边界，不能凭经验假设。新代码的层级归属必须与架构文档一致。

### 范式优先
1. 找到与当前任务功能相似的已有模块
2. 阅读其文件组织、命名、方法签名、错误处理方式
3. 新代码与它们保持一致
4. 如果没有类似范式 → 以简单为先

### 层级决策
```
这段代码做的事是？
├─ URL/HTTP/参数？          → 接口/接入层
├─ 能不能做/要不要做/顺序？ → 业务/编排层
├─ 拼SQL/操作DB？           → 数据/持久层
├─ 纯计算/统计/数据转换？   → 独立能力层
├─ 对接外部API/格式转换？   → 独立能力层
├─ 横切关注点（日志/缓存）？ → 独立工具/中间件
└─ 说不清？                 → 停下来
```

### 跨层污染模式
| 污染 | 表现 | 修正 |
|------|------|------|
| 数据上移 | 业务层拼数据查询语句 | 下沉到数据层 |
| 业务下移 | 数据层含业务条件判断 | 上移到业务层 |
| 接口膨胀 | 接口层写大量业务逻辑 | 下沉到业务层 |
| 借道访问 | 通过不相关模块获取底层资源 | 使用自己模块的依赖 |
| 跨层跳跃 | 接口层直接调数据层 | 插入业务层 |
| 计算混入 | 业务/数据层写大量计算 | 抽取到计算层 |

**输出**：改动清单，每个文件/函数标注层级归属

```markdown
| 文件 | 改动类型 | 层级 | 改动说明 |
|------|---------|------|---------|
| mapper/x.py | 新增方法 | 数据层 | add get_xxx() |
| service/y.py | 修改编排 | 业务层 | 调用新方法 |
| api/z.py | 新增接口 | 接口层 | GET /xxx |
```

## Pass 4：执行改动

**目标**：精准改动，不留垃圾。

1. 严格按 Pass 3 的改动清单执行，不改清单外的文件
2. 范式一致——新代码的风格与 Pass 2 找到的范式参考一致
3. **清理纪律（关卡 4）**：废弃字段/方法/导入直接删除，不留兼容桥、墓碑注释、fallback 兜底
4. 不改"顺手"看到的无关问题——那些应该记下来，下次单独处理

---

## Pass 5：自检（关卡 5）— 关键

**目标**：在交付前，用审查的标准检查自己。自己发现的问题修起来代价最低。

1. **先重读 `06-开发规范/`**，提取可逐条验证的条款——不凭记忆，打开文档逐条对照
2. 按[代码审查](代码审查.md) Pass 2 的 5 个维度，逐条检查本次改动：
   - 维度 1 — 架构合规：分层对了吗？有没有借道？通用组件污染？
   - 维度 2 — 代码质量：重复？魔法值？兼容垃圾？值语义？类型完整？
   - 维度 3 — 性能与数据：N+1？无界扫描？并发安全？热路径缓存？
   - 维度 4 — 测试覆盖：改了的有测试吗？是真实链路吗？
   - 维度 5 — 前端专项（如涉及）：状态管理？竞态？防重？
3. 每项输出具体结果，不得笼统说"通过"
4. 违规 → 修复 → 重新检查 → 循环直到零违规，最多 3 轮

---

## Pass 6：文档同步（关卡 6）— 硬性清单

**目标**：代码落盘不是终点，文档跟着代码一起变才是终点。

**必须逐条确认，缺一不可：**

```
☐ 10-开发计划/    开发计划已落盘？
☐ 09-AI开发日志/  本次改动已记录？（每次代码落盘必须记）
☐ 04-功能设计/    功能/API/数据模型有变化 → 蓝图已同步到最新设计？
☐ 05-数据库/      DDL/表结构有变更 → SQL 脚本已更新？
☐ 03-系统架构/    分层/模块/技术选型有变化 → 架构文档已更新？
☐ 11-测试计划/    >3 用例 → 测试计划已落盘？
☐ 测试报告        测试已跑 → 报告已落盘？
```

**任一条未完成 → 阻塞，补完才过。** 禁止口头说"已更新"而不实际落盘。

特别强调：
- **`04-功能设计/` 不是事后补**：功能变更后，设计文档必须同步到最新设计。这是蓝图，代码照着它施工。
- **`09-AI开发日志/` 每次代码落盘必须记**：不是"有空再记"，是代码改动落地后立即记录。

---

## 重构专项要求

1. **先理解旧架构**：为什么当初这样设计？改了会影响谁？
2. **列出所有下游依赖**：全部找完才能动手
3. **重构 = 先清理再新增**：废弃代码删，旧逻辑搬，不兼容
4. **不分步"慢慢改"**：同一个重构任务不留过渡态

---

## 禁止行为

- 禁止看入口 + 几个引用就说"已全面了解"
- 禁止跳过基类/父类直接看子类
- 禁止基于文件名猜测角色而不打开阅读
- 禁止凭训练数据经验写代码
- 禁止未确认框架版本和源码就动手
- 禁止修改子类而不读父类/基类/接口
- 禁止"顺手"做未要求的功能或重构
- 禁止保留废弃代码、墓碑注释、兼容代码
- 禁止创建无业务价值的抽象层
- 禁止跳过 Pass 1——没有设计方案就开始写代码
- 禁止把开发计划写到开发日志里——计划是 `10-开发计划/`，日志是 `09-AI开发日志/`
- 禁止跳过 Pass 5 自检——自己都懒得查的问题，等着审查来抓
- 禁止跳过 Pass 6 文档同步——代码改了文档不改等于没做完
- 禁止功能/API/数据模型变更后不更新 `04-功能设计/`——蓝图必须是最新的
