# 测试流程

适用场景：编写测试、执行测试、验证功能正确性。

---

## 全局锁：测试流程必须从头到尾顺序执行，不得跳过任何 Pass。

---

## Pass 0：测试计划

**目标**：先写测试计划，再动手测。和开发计划一样，测试计划是 pre-test 设计文档。

### 强制规则

1. **在进入 Pass 1 之前**，必须先输出测试计划文档
2. 测试计划落盘后，才能继续
3. 小改动（单文件、纯函数、已有复用测试）可跳过 Pass 0，直接在日志中简要说明测试策略

### 计划内容要求

使用 `assets/project-template/11-测试计划/测试计划模板.md` 模板，至少包含：

- 背景与目标
- 测试范围（纳入/不纳入及原因）
- 测试策略（方法、环境、数据）
- 测试用例概要
- 风险与假设

### 文档落盘

- **存放位置**：`11-测试计划/YYYY-MM-DD/测试计划_功能名.md`
- 按日期组织，命名规则与 `10-开发计划/` 一致
- 落盘后进入 Pass 1

---

## Pass 1：确定测试范围

**目标**：知道测什么、为什么测、测到什么程度算够。

1. 明确本次改动的**影响范围**：哪些模块、哪些函数、哪些接口
2. 从项目规范中提取测试相关条款（如有）
3. 逐项判断需要的测试级别：

| 改动类型 | 最低测试要求 |
|---------|------------|
| 新增数据层方法 | 至少一条集成测试，验证查询正确性 |
| 新增/修改业务逻辑 | 测试核心编排路径（正常 + 关键错误） |
| 新增接口 | 测试正常返回 + 关键错误路径 + 边界值 |
| 数据量/阈值校验逻辑 | 测试边界条件：不足、刚好、超出 |
| 修复 Bug | 先写能复现 Bug 的测试（确认 Bug 存在），修完后测试通过 |
| 小修复/格式调整 | 确认现有测试仍然通过 |

4. 列出本次**不测什么**并说明原因（如：已有覆盖、后续补、不适用）

**输出**：测试范围清单 + 不覆盖项及原因

---

## Pass 2：编写测试

**目标**：写能验证真实链路的测试，不是只测纯函数。

### 核心原则：验证真实链路

纯函数测试只是起点。不能 mock 掉核心依赖后声称"测试通过"：

| 必须走真实链路 | 说明 |
|--------------|------|
| 数据库查询 | 真实 DB 或测试 DB，不 mock SQL 执行 |
| 业务编排 | 注入真实依赖（非 mock），验证完整调用链 |
| 定时/后台任务 | 触发执行，验证副作用和数据落盘 |
| 接口响应 | 发真实 HTTP 请求，验证状态码和响应结构 |

### 测试文件组织

```
tests/
├── {module}/
│   ├── test_{feature}.ext       # 功能测试
│   └── conftest.ext             # 该模块 fixtures
```

与源码模块一一对应，命名 `test_` 前缀。

### 关注点

- **边界条件优先**：空值、零值、极限值、空列表、null、超时
- **错误路径覆盖**：不只是 happy path，也要测异常处理
- **数据隔离**：测试数据不与生产数据混淆，测试后清理

---

## Pass 3：运行与验证

**目标**：确认测试全部通过，失败则定位根因。

1. 运行全部相关测试（不只看新增的，也跑已有的确认没被改坏）
2. 测试失败 → **先分析根因，不要跳过或注释掉失败的测试**
   - 是测试写错了？→ 修测试
   - 是代码有问题？→ 修代码
   - 是环境和数据导致？→ 确认后调整
3. 全部通过后，确认覆盖了 Pass 1 列出的所有要求

---

## Pass 4：覆盖率检查与补漏

**目标**：确认没有漏掉关键链路。

对照 Pass 1 的测试范围清单，逐项检查：

1. 每个改动点都有对应的测试吗？
2. 每条链路都至少走了一遍吗？
3. 每个边界条件都覆盖了吗？
4. 有没有只测了纯函数、mock 了核心依赖就跳过的？

如果发现漏测 → 回到 Pass 2 补充，然后 Pass 3 重新验证。

---

## Pass 5：输出测试报告并落盘（关卡 6 强制项）

**目标**：测试跑完不是终点，报告落盘才是。

1. 按测试报告格式输出完整报告
2. **落盘到 `11-测试计划/{日期}/`**，与测试计划放在同一日期目录
3. 测试报告必须包含：范围、结果、覆盖链路、边界条件、未覆盖项及原因
4. 报告落盘后，在关卡 6 文档同步清单的「测试报告」项打勾

---

## 测试报告

测试完成后必须输出报告：

```markdown
## 测试报告

### 测试范围
- 涉及模块：xx, yy
- 测试文件：N 个

### 测试结果
- 通过：X / 总计：Y

### 覆盖链路
| 链路 | 测试文件 | 是否通过 |
|------|---------|---------|
| get_table_health 完整查询 | test_xxx.py | ✓ |
| Dashboard 快照读取 | test_yyy.py | ✓ |

### 覆盖边界条件
- 正常路径 ✓
- 空数据 ✓
- 边界值 ✓
- 异常路径 ✓

### 未覆盖项
| 项目 | 原因 |
|------|------|
| xx 接口错误路径 | 下轮补充 |
```

---

## 禁止行为

- 禁止跳过 Pass 0——没有测试计划就直接写测试
- 禁止只测纯函数，不测真实链路
- 禁止 mock 核心依赖后声称"测试通过"
- 禁止跳过边界条件测试
- 禁止把失败的测试注释掉或 skip 掉而不分析根因
- 禁止测试完成后不落盘报告，笼统说"测试过了"
- 禁止测试报告不放 `11-测试计划/` 目录——报告和计划放一起
- 禁止测试数据污染生产环境
