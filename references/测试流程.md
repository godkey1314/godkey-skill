# 测试流程

适用场景：编写测试、执行测试、验证功能正确性。

---

## 核心原则：必须验证真实链路

纯函数测试只是起点。审查不能只看纯函数测试通过就放行：

| 必须测试 | 说明 |
|---------|------|
| 数据库查询 | 真实数据库或测试数据库，不 mock SQL 执行 |
| 业务编排 | 注入真实依赖，验证完整调用链 |
| 定时任务/后台任务 | 触发执行，验证副作用和数据落盘 |
| 接口响应 | 发真实 HTTP 请求，验证状态码和响应结构 |

---

## 测试覆盖最低要求

- 新增数据层方法 → 至少一条集成测试验证查询正确性
- 新增业务逻辑 → 测试核心编排路径
- 新增接口 → 测试正常返回和关键错误路径
- 数据量校验逻辑 → 测试边界条件（数量不足、刚好达标、超出）

---

## 测试文件组织

```
tests/
├── {module}/
│   ├── test_{feature}.ext       # 功能测试
│   └── conftest.ext             # 该模块的 fixtures（如有）
```

测试文件与源码模块一一对应，命名 `test_` 前缀。

---

## 测试执行流程

1. 确认测试范围和覆盖要求
2. 编写/更新测试代码
3. 运行测试，查看结果
4. 失败 → 分析根因 → 修复 → 重新运行
5. 全部通过 → 输出测试报告（覆盖了哪些链路、哪些边界条件）

---

## 禁止行为

- 禁止只测纯函数，不测真实链路
- 禁止 mock 核心依赖后声称"测试通过"
- 禁止跳过边界条件测试
