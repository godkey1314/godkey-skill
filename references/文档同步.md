# 文档同步

适用场景：贯穿整个开发流程，不是只在最后执行一次，而是在每个关键节点都有对应的文档动作。

---

## 文档目录初始化（前置关卡执行，任何代码任务第一步）

### 目录位置规则

- **单体项目**（一个代码仓库、一个交付物）：文档放在项目根目录下的 `doc/`
- **多模块项目**（前后端分离、多子仓库/子模块、多个独立服务）：文档放在仓库根目录下的 `{project_name}_doc/`，如 `ai_stock_doc/`、`shop_doc/`

如果拿不准算单模块还是多模块 → 问用户。

### 初始化流程（前置关卡执行）

初始化不是机械复制，而是分级处理：

```
全新项目（doc/ 完全不存在）
  → 从 project-template/ 复制完整骨架
  → AI 逐项提问，用户主导填充（产品概述、技术选型、架构）
  → 用户说"够了"就停

部分缺失
  → 报告缺失项给用户
  → 缺失目录从模板补充结构（不填内容）
  → 用户决定哪些现在补、哪些后面按需补

文档完整
  → 直接通过
```

**关键原则**：文档是用户的文档，AI 提问、用户回答、AI 落盘。AI 不替用户做产品决策。够了就停，不强制填满。

### 标准目录结构

```
doc/ 或 {project_name}_doc/
├── 01-产品介绍/        # 产品概述、快速开始、核心特性
├── 02-使用指南/        # 用户操作手册
├── 03-系统架构/        # 架构设计、技术选型、数据库设计概要
├── 04-功能设计/        # 各模块/API 的功能设计文档
├── 05-数据库/          # SQL 脚本、表结构、迁移记录
├── 06-开发规范/        # 代码规范、Git 规范
├── 07-部署运维/        # 环境配置、部署手册
├── 08-资源文件/        # API 文档、图片等
├── 09-AI开发日志/      # AI 对话日志（每次代码落盘记）
├── 10-开发计划/        # 开发方案、审查报告
└── 11-测试计划/        # 测试计划、测试报告
```

## 全场景文档触发表

**每个开发节点都有对应的文档动作。任务开始时先读文档了解上下文，任务结束时写文档留痕。**

| 节点 | 触发时机 | 文档动作 |
|------|---------|---------|
| **文档体系检查** | **任何代码任务第一步**（前置关卡） | 检查 doc/ 是否存在，缺失则用模板初始化；逐目录审计，输出状态清单 |
| **开始工作** | 每次对话 / 新任务开始 | 读 `09-AI开发日志/` 了解历史，读 `04-功能设计/` 了解蓝图，读 `10-开发计划/` 了解当前计划 |
| **设计方案** | 新功能（>3 文件）、重构、架构调整、专项修复 | 先读 `04-功能设计/` 对应模块，再写方案到 `10-开发计划/{日期}/` |
| **制定测试计划** | 编写测试（>3 用例 或 涉及集成/端到端测试） | 创建 `11-测试计划/{日期}/` 下的测试计划文档 |
| **代码落盘** | 任何代码文件发生变动 | 写 `09-AI开发日志/日志/` 日志（必须，不可跳过） |
| **数据库变更** | 加表、改字段、改索引、数据迁移 | 更新 `05-数据库/` 对应 SQL 文件或迁移脚本 |
| **功能开发完成** | 新功能或重构的代码已合并 | **强制更新 `04-功能设计/`** 对应模块文档，同步 API/表结构/业务流程。设计文档代表最新设计，代码照着它施工。 |
| **架构调整** | 分层结构、技术选型、模块边界变更 | 更新 `03-系统架构/` 架构设计 + 技术选型文档 |
| **代码审查完成** | 审查流程 Pass 4 结束后 | 输出审查报告到 `10-开发计划/{日期}/`，文件名含"审查"标识 |
| **测试完成** | 测试流程结束后 | 输出测试报告到 `11-测试计划/{日期}/`，与测试计划放在一起 |
| **部署/环境变更** | 首次部署、环境迁移、配置变更、新依赖 | 更新 `07-部署运维/` 环境配置或部署手册 |
| **产品定位变更** | 产品方向、目标用户、核心特性变化 | 更新 `01-产品介绍/` |
| **用户流程变更** | 界面操作流程、使用方式变化 | 更新 `02-使用指南/` |
| **开发规范变更** | 编码规范、Git 规范调整 | 更新 `06-开发规范/` |

## 各文档详细规范

### 09-AI开发日志（每次代码落盘必须记）

**存放位置**：`09-AI开发日志/日志/`

**命名**：`YYYYMMDD_SessionNNN_任务名.md`（编号接续已有日志）

**必须包含**：做了什么、变更前快照、涉及文件、问题与决策、下次计划、遗留风险

**规则**：
- 同一连续任务不重复开新日志——追问、返工继续更新原日志
- 只有切换到新任务时才创建新日志
- 任务开始前先读最近 2-3 篇日志，了解上下文

### 10-开发计划

**存放位置**：`10-开发计划/YYYY-MM-DD/方案名称.md`（当天多个加 `_01/_02` 后缀）

**何时创建**：新功能（>3 文件）、重构（跨模块）、架构调整、专项修复方案

**何时不创建**：单文件小修、文案调整、配置修改、格式修正

**内容要求**：背景与目标、涉及范围、实现方案、影响分析、测试要点

**审查报告**：审查完成后，也放在此处，文件名含"审查"或"修复清单"标识。格式见[代码审查](代码审查.md)的"审查报告格式"章节。

### 11-测试计划

**存放位置**：`11-测试计划/`

**何时创建**：编写测试（>3 用例或涉及集成/端到端测试）。小改动（纯函数、已有复用测试）可跳过，在日志中简述策略。

**内容要求**：背景与目标、测试范围（纳入/不纳入）、测试策略、测试用例概要、风险与假设。使用 `测试计划模板.md`。

**测试报告**：测试完成后输出到同一日期目录，包含测试范围、覆盖链路、边界条件、通过/失败结果、未覆盖项及原因。

### 04-功能设计

**何时更新**：功能开发完成、代码合并后，对照实际代码同步

**更新内容**：API 路径/入参/出参、数据库表结构、业务流程图、关键设计决策

**不做什么**：不在开发过程中频繁修改——过程变更记在开发日志。方案与最终实现的偏离在日志中说明。

### 03-系统架构

**何时更新**：分层结构变化、模块新增/合并/拆分、技术栈升级、第三方服务切换

**更新内容**：分层架构图、模块依赖关系、技术选型理由、安全/性能设计

### 05-数据库

**何时更新**：加表、改字段、加索引、数据迁移、初始化数据变更

**更新内容**：DDL 脚本（`05-数据库/sql/`）、表结构文档、索引说明、迁移步骤

### 07-部署运维

**何时更新**：首次部署、环境变量变更、新增中间件/服务、配置项调整、Docker/Nginx 变更

**更新内容**：环境配置清单、部署步骤、回滚方案

### 01-产品介绍、02-使用指南、06-开发规范、08-资源文件

按需更新，触发条件明确时不遗漏。

---

## 禁止行为

- 禁止代码落盘后不记日志（最基础要求）
- 禁止审查/测试完成后只有口头结论，不落盘报告
- 禁止文档与代码不一致——代码已改文档还写旧的
- 禁止无脑建文档——改一行就建一个开发计划
- 禁止只写不维护的僵尸文档
- 禁止任务开始时跳过读已有日志和计划——上下文断了要重建
