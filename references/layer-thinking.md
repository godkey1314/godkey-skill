# 层级思维：代码该放哪一层

本文档定义了一个跨语言、跨框架的通用分层决策框架。当你不确定代码应该放在哪个文件/模块时，按此决策树判断。

---

## 通用四层模型

不管你用什么语言，大多数后端/业务系统都可以映射为这四层：

```
┌──────────────────────────────────┐
│  A. 接口/表现层                    │  路由、入参验证、响应包装、调用下层
│     Controller / Handler / Route  │
├──────────────────────────────────┤
│  B. 业务/编排层                    │  业务判断、流程编排、事务边界、权限
│     Service / UseCase / Domain    │
├──────────────────────────────────┤
│  C. 数据/持久层                    │  数据库访问、查询构建、ORM 操作
│     Mapper / Repository / DAO     │
├──────────────────────────────────┤
│  D. 独立能力层（可选）               │  纯计算、外部 API 封装、AI 代理
│     Calculator / Provider / Agent │
└──────────────────────────────────┘
```

## 决策树：新增代码放哪

当你需要写一段逻辑时，按以下顺序判断：

```
这段代码做的事是？
├─ 定义 URL 路径、HTTP 方法、参数来源？
│   → A 层（接口层）
│
├─ 判断"能不能做""要不要做""先后顺序"？
│   → B 层（业务层）
│
├─ 拼 SQL、操作数据库、定义 ORM 模型？
│   → C 层（数据层）
│
├─ 纯数学计算、统计、数据转换，不涉及业务判断？
│   → D 层（独立计算层）
│
├─ 对接外部 API，做数据格式转换？
│   → D 层（独立能力层，如 Provider/Gateway）
│
├─ AI 推理、分析、生成？调用 AI 模型？
│   → D 层（独立能力层，如 Agent/Skill）
│
├─ 横切关注点（日志、缓存、权限）？
│   → 独立工具/中间件/装饰器，不放入业务层
│
└─ 说不清属于哪一层？
    → 先停下来，重新理解架构再决定
```

## 跨层污染模式（必查清单）

以下每一种都是"代码放错了位置"，发现即需修正：

| 污染模式 | 表现 | 修正 |
|---------|------|------|
| **SQL 上移** | Service/业务层中出现 SQL 字符串拼接 | SQL 下沉到 Mapper/Repository |
| **业务下移** | Mapper/数据层中出现业务条件判断 | 业务逻辑上移到 Service |
| **接口膨胀** | Controller 中写大量业务处理逻辑 | 业务逻辑下沉到 Service |
| **借道访问** | Service 通过不相关的其他 Mapper 获取数据库会话 | 使用自己模块的 Mapper 或注入正确的依赖 |
| **跨层跳跃** | Controller 直接调用 Mapper，跳过 Service | 中间插入 Service 层 |
| **计算混入** | 在 Service 或 Mapper 中写大量数据计算/统计逻辑 | 抽取到独立计算层 |

## 实际案例

### 案例：数据质量检查

你需要统计多张业务表的数据量、字段空值率和数值违规情况。

**错误做法（跨层污染）**：

在 Service 中直接拼接 SQL，并借用另一个不相关 Mapper 的数据库会话，逐表、逐字段、逐规则各扫一次。

问题：
- SQL 上移 → SQL 应该在数据层，不在业务层
- 借道访问 → 不应该通过其他不相关模块间接获取数据库会话
- 逐条查询 → 对 N 张表的 M 个字段逐一执行聚合查询，产生 N×M 次数据库请求

**正确做法**：

1. 新建一个数据层组件，集中承载所有质量检查所需的查询
2. 业务层只做三件事：读取检查规则 → 调用数据层获取结果 → 组装状态摘要
3. 一次 SQL 统计多个字段（如 `COUNT(*) FILTER (WHERE col IS NULL)`），一次查询完成多规则检查
4. 该模块自己管理自己的数据库会话，独立于其他模块

```
接口层:    GET /dashboard → 调用 Service
业务层:    读取规则 → 调用 Mapper → 组装结果
数据层:    SELECT date, COUNT(*) ... GROUP BY date
```

### 案例：废弃字段的兼容代码

某个字段 `old_score` 已不再被接口返回，但代码中保留了：

```python
score = getattr(row, "old_score", 0)  # 兼容旧数据
```

**判断**：字段已废弃 → 删除代码，删除模型中的字段定义，如有数据库列也排入清理计划。

**不要做**：保留代码 + 加注释"待删除"。要么删，要么不删，没有"标记"状态。

### 案例：计数查询加载全量对象

需要知道有效记录总数，代码加载了所有完整对象再取 `len()`。

```python
# 错误：加载全量对象
records = await service.find_by_filters(status="active")
count = len(records)

# 正确：直接 COUNT
count = await repository.count_by_filters(status="active")
```

**判断**：数据层的查询优化是数据层的职责，不能因为"方便"就拉全量数据到业务层。
