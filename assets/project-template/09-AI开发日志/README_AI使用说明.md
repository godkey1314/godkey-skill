# AI使用说明

> **📖 AI开发者必读文档**

---

## 🎯 你是谁

你是一个AI开发助手，负责协助完成本项目的开发工作。

本文档是专门为你准备的操作指南，请**每次开发会话开始时仔细阅读**。

---

## 📂 文档体系说明

本项目采用**模板驱动**的文档管理方式：

```
ocra-doc/
├── 01-产品介绍/        产品概述、快速开始
├── 02-使用指南/        用户手册（按模块）
├── 03-系统架构/        架构设计、技术选型、数据库设计
├── 04-功能设计/        功能详细设计（核心！）
├── 05-数据库/          SQL文件
├── 06-开发规范/        代码规范、Git规范
├── 07-部署运维/        部署和运维文档
├── 08-资源文件/        Swagger、图片等
└── 09-AI开发日志/      你的工作记录（本目录）
```

## 🔄 你的工作流程

### 第1步：每次会话开始时（必做）

**阅读以下文档（按顺序）**：

1. **本文件**（`README_AI使用说明.md`）
   - 了解工作流程

2. **最近3-5个开发日志**（`日志/` 目录）
   - 了解最近做了什么
   - 了解当前项目状态
   - 了解遇到的问题

3. **相关功能设计文档**（`04-功能设计/`）
   - 了解本次要开发的功能详情

**阅读时间**：约5-10分钟

---

### 第2步：确认本次任务

与用户确认本次要完成的任务：
- 要开发什么功能？
- 涉及哪些模块？
- 有什么特殊要求？

---

### 第3步：开发过程中

#### 如果是新功能开发
1. 先编写/完善功能设计文档
   - 路径：`04-功能设计/[模块名]设计.md`
   - 使用模板：`04-功能设计/模块设计模板.md`
   
2. 如果需要新建数据库表
   - 创建SQL文件：`05-数据库/sql/[日期]_[表名].sql`
   
3. 开发代码
   - 遵循 `06-开发规范/` 中的规范

#### 如果是修改现有功能
1. 更新功能设计文档（在文档末尾追加"变更记录"）
2. 修改代码
3. 如有数据库变更，创建新的SQL文件

---

### 第4步：对话结束时（强制）

**必须创建本次开发日志**：

1. 在 `日志/` 目录创建新文件
2. 文件命名：`[日期]_Session[编号]_[任务名].md`
3. 示例：`20251027_Session001_用户登录开发.md`
4. 使用模板：`日志模板.md`

**日志内容必须包含**：
- 本次完成了什么
- 创建/修改了哪些文档
- 创建/修改了哪些代码文件
- 遇到的问题和解决方案
- 下次要做什么

---

## 📝 文档更新规则

### 规则1：能追加就追加，不修改旧内容
```markdown
## 变更记录

### v1.1 - 2025-10-28
- 新增手机号登录方式
- 调整登录流程

### v1.0 - 2025-10-27
- 初始版本
```

### 规则2：新文档单独创建
```
不要修改：20251027_用户登录.md
而是创建：20251028_用户登录_补充手机号.md
```

### 规则3：文件命名包含关键信息
```
✅ 好的命名：
20251027_Session001_用户登录功能.md
- 日期：20251027
- 编号：Session001
- 任务：用户登录功能

❌ 不好的命名：
日志1.md
```

## 🎯 工作重点

### 最重要的是：开发日志
**每次对话结束必须创建日志！**

原因：
- 下次AI接手时靠日志理解项目状态
- 日志是上下文连续性的关键
- 不写日志 = 信息丢失

### 其次重要的是：功能设计文档
在开发前/开发中完善功能设计文档，确保：
- 功能需求清晰
- 数据结构明确
- 接口定义完整

---

## ⚠️ 注意事项

### 1. 不要假设
如果不确定某个需求，直接问用户，不要自己猜测。

### 2. 不要过度设计
只实现当前需求，不要提前实现未来可能需要的功能。

### 3. 不要忽略文档
**写代码的同时必须更新文档**，不要事后补。

### 4. 不要硬编码
配置信息、密钥等必须使用环境变量。

### 5. 不要忽略规范
遵循 `06-开发规范/` 中的代码规范和Git规范。

---

## 📋 开发完成检查清单

每次对话结束前，检查以下项目：

- [ ] 功能代码已完成
- [ ] 功能设计文档已更新/创建
- [ ] 如有数据库变更，SQL文件已创建
- [ ] 代码符合规范（命名、注释、格式）
- [ ] 创建了本次开发日志
- [ ] 日志中记录了所有关键信息
- [ ] Git commit message 符合规范

---

## 🤝 与用户沟通

### 汇报进度
主动告诉用户当前进展：
- "我已经完成了XX功能"
- "我创建了XX文档"
- "遇到了XX问题，已解决/需要确认"

### 寻求确认
不确定时及时问用户：
- "这个功能是否应该XX？"
- "数据库字段XX的类型应该是XX吗？"
- "接口返回格式是否应该是XX？"

### 给出建议
基于经验给出专业建议：
- "建议使用XX方案，因为XX"
- "这里可能存在XX风险，建议XX"

---

## 💡 开发技巧

### 1. 先设计后开发
不要一上来就写代码，先完善功能设计文档。

### 2. 小步快跑
把大任务拆分成小任务，逐步完成。

### 3. 及时测试
写完代码立即测试，不要积累问题。

### 4. 记录问题
遇到问题和解决方案都记录在日志中。
