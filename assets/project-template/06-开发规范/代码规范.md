# 代码规范

> 本项目代码编写规范。以下通用原则跨语言适用，具体实现按各语言惯例调整。
> 当项目有自己的规范文档时，以项目规范为准；项目无规范时，以下为默认标准。

---

## 一、范式优先

### 1.1 范式高于经验

在项目中写任何代码前，必须先找到项目中已有的类似实现作为参考。代码风格、架构决策、工具选型以项目现有范式为准，不以个人偏好为准。

```
接到任务 → 找项目中类似功能 → 理解它的组织方式和代码风格 → 按同样范式写
```

而不是：

```
接到任务 → 凭经验写 → 风格和项目不一致
```

### 1.2 范式查找方法

1. 找到与当前任务功能相似的已有模块
2. 阅读该模块的文件组织、类命名、方法签名、错误处理方式
3. 新代码的风格与它们保持一致
4. 如果项目中没有类似范式 → 参考项目规范文档；规范文档也没有 → 以简单为先

---

## 二、分层纪律

### 2.1 每一行代码都有层级归属

任何代码都存在于明确的架构层中。写代码前必须先确定：这一行属于哪一层？

每一层只能向下依赖，不能反向。通用三层模型：

| 层 | 职责 | 不能做 |
|---|---|---|
| **接入层** | 接收外部请求/事件，参数校验，调用业务层，返回响应 | 写业务判断、直接访问持久化设施 |
| **业务层** | 业务逻辑、流程编排、事务边界、跨模块协作 | 直接操作持久化设施的实现细节 |
| **基础层** | 数据持久化、外部服务调用、消息队列、文件系统访问 | 写业务判断、做业务过滤 |

### 2.2 独立能力层

除了通用三层，项目可能有独立的能力层：

- **计算层**：纯数学计算、统计聚合、数据转换（不写业务判断）
- **代理层**：对接外部 API、数据格式转换（不写业务判断）
- **智能层**：AI 推理、分析、生成

### 2.3 跨层污染识别

以下信号表示代码位置放错了（语言无关）：

- 在业务层中看到拼接的数据查询语句 → 应在基础层
- 在基础层中看到业务条件判断 → 应在业务层
- 在接入层中看到持久化操作 → 应在业务层/基础层
- 借用不相关模块的底层资源（连接、会话、句柄）→ 抽象错误
- 在能力层中硬编码业务决策 → 应在上层完成

## 三、模块边界

### 3.1 何时新建模块

满足以下任一条件，应新建独立模块：

- 有独立的业务领域（和现有模块不在同一领域）
- 有自己的数据模型
- 会被其他多个模块复用
- 有独立的生命周期或部署节奏

### 3.2 何时扩展现有模块

- 处理对象和已有模块相同
- 只是已有功能的补充或延伸
- 和已有模块共享数据模型

### 3.3 不做什么

- 不为单一场景创建抽象层
- 不创建无业务价值的中间层
- 不为"将来可能需要"预留结构

---

## 四、代码清洁度

### 4.1 删除高于保留

- 废弃字段 → 删除，不用 fallback 兜底
- 不再调用的方法 → 删除
- 不再使用的导入 → 删除
- 墓碑注释（"已移除""待删除""临时方案"）→ 直接处理，不留标记

### 4.2 不留兼容垃圾

- 接口不再返回的字段 → 从模型和序列化中删除
- 历史兼容逻辑 → 如果历史数据已清理，兼容逻辑删除
- "以防万一"的兜底 → 不需要

### 4.3 禁止魔法值

状态码、类型标识、业务常量必须使用枚举或命名常量，不得出现裸数字/字符串：

```python
# 错误：魔法值
if record.deleted == 0:        # 0 代表什么？
    record.status = "active"    # 还有哪些可能值？

# 正确：枚举/常量
if record.deleted == DeleteStatus.NORMAL:
    record.status = Status.ACTIVE
```

所有有业务含义的固定值集合都必须定义为枚举。包括但不限于：删除标记、状态、类型、模式、来源、级别。

### 4.4 模型文件组织

每种模型独立成文件，禁止将不同类型模型混在同一文件：

```
# 正确
models/
├── entity/user.py       # 数据实体
├── entity/order.py      # 另一个实体 → 不同文件
├── api/user_result.py   # 接口视图对象
└── api/order_result.py

# 错误
models.py                # 所有模型塞一个文件
```

- 每个文件只放一种模型/一个实体
- 数据实体和接口视图对象物理隔离

### 4.5 注释原则

- 只在业务边界、复杂决策、非显然逻辑处写注释
- 代码自解释优于注释解释
- 禁止无意义注释（如 `# 获取数据` 在 `get_data()` 之前）

## 五、反过度工程

### 5.1 不做的事

- 不做用户未要求的功能
- 不加不会发生的场景的错误处理
- 不为一次性的操作创建工具函数
- 不引入不使用的依赖

### 5.2 简单优先

- 三个相似代码片段 > 一个过早抽象
- 直接调用 > 多层委托
- 如无必要，勿增实体

---

## 六、依赖方向

### 6.1 单向依赖

依赖关系应该单向流动。底层不依赖上层，同层可互相调用但避免循环。

### 6.2 不借道

- 不通过不相关的模块间接获取资源（如借用其他模块的数据库会话）
- 每个模块的资源由模块自身管理

---

## 七、数据与 I/O 访问纪律

### 7.1 查询归属

- 所有持久化操作只能在基础层（数据访问组件）
- 其他层通过调用基础层接口获取数据，不直接操作数据库、文件系统或外部服务

### 7.2 性能铁律

| 禁止 | 替代做法 |
|------|---------|
| 加载完整对象列表只为了取数量 | 使用计数查询 |
| 对 N 个目标逐条查询（N+1） | 合并为批量查询或一次查询 |
| 无界并发（N 个任务同时启动无上限） | 使用并发限制（如信号量，建议 3-5） |
| 同一个异步会话/连接被多协程/线程共享 | 各自独立获取会话，或改为顺序执行 |

### 7.3 查询范围

- 明确指定查询范围（时间、分页等），不无界扫描
- 聚合统计按合理粒度分组，不做全局聚合稀释局部问题

### 7.4 热路径缓存

高频访问的查询结果应优先使用预计算结果：

| 场景 | 做法 |
|------|------|
| 首页/Dashboard 接口 | 优先读快照/缓存，实时计算作为 fallback |
| 批处理产出的结果 | 持久化结果，接口读结果而非重新计算 |
| 多次请求相同数据 | 缓存，设置合理过期时间 |

### 7.5 查询合并

对同一数据源的多个统计查询必须合并，禁止逐字段/逐条件分别查询。一次查询统计全部所需字段。

---

## 八、代码重复与抽象时机

### 8.1 重复信号

审查代码时，横向对比同类文件。如果发现以下信号，说明存在重复：

- 3 个以上文件有**结构相同、仅参数不同**的方法
- 同目录下多个文件的**方法签名和实现逻辑一致**
- 同样的模式在多处各写一遍

### 8.2 抽象时机

| 重复次数 | 做法 |
|---------|------|
| 1-2 次 | 保留重复，不抽象 |
| 3-5 次 | 抽取公共方法/函数 |
| 5 次以上 | 抽取 Mixin / 基类 / 工具模块 |

### 8.3 抽象方式

- 同一模块内重复 → 私有函数
- 同层多个模块重复 → 抽象基类 / Mixin / Helper
- 跨层重复 → 独立工具模块

原则：**三个相似 > 一个过早抽象，但十五个重复 > 必须抽象。**

## 九、值语义陷阱

### 9.1 零值与 None 不可混用

`0`、空字符串 `""`、空列表 `[]` 和 `None` 语义不同。禁止用 falsy 检查替代 None 检查：

```python
# 错误：0 被当成"不存在"
value = result.get("rate")
rate = to_number(value) if value else null  # rate=0 → null！

# 正确
rate = to_number(value) if value is not null else null
```

### 9.2 各语言检查方式

| 语言 | 区分方式 |
|------|---------|
| Python | `if x is not None` 而非 `if x` |
| JavaScript/TS | `x != null` 而非 `!x` |
| Go | 显式检查零值 vs nil |
| Java/Kotlin | `Optional` 或显式 null 检查 |
| Rust | `Option<T>` 模式匹配 |
| C# | `x is not null` 或 `Nullable<T>.HasValue` |

规则：**区分"值为零"和"值不存在"。**

## 十、通用组件职责边界

### 10.1 通用组件不感知业务语义

通用组件（如 TaskExecutor、框架中间件）不应包含特定业务领域的判断逻辑：

```python
# 错误：通用执行器识别特定业务告警类型
if warning.type == "low_inventory":
    handle_inventory_alert()

# 正确：通用执行器只记录通用结构，业务处理留给业务模块
warnings = [{"type": t, "context": {...}}]
```

### 10.2 修正方式

- 通用组件只定义通用数据结构
- 业务语义的解析和处理放在业务模块中
- 不要为了一个业务场景修改通用组件

## 十一、测试纪律

### 11.1 测试必须验证真实链路

纯函数测试只是起点。代码审查不能只看纯函数测试通过就放行：

| 必须测试 | 说明 |
|---------|------|
| 数据库查询 | 真实数据库或测试数据库，不 mock 掉 SQL 执行 |
| 业务编排 | 注入真实依赖，验证完整调用链 |
| 定时任务 | 触发任务执行，验证数据落盘和副作用 |
| 接口响应 | 发真实 HTTP 请求，验证状态码和响应结构 |

### 11.2 测试覆盖的最低要求

- 新增数据层方法 → 至少一条集成测试验证 SQL 正确性
- 新增业务逻辑 → 测试核心编排路径
- 新增接口 → 测试正常返回和关键错误路径
- 数据量校验逻辑 → 测试边界条件（数量不足、刚好达标、超出）

### 11.3 测试文件组织

```
tests/
├── {module}/
│   ├── test_{feature}.ext       # 功能测试
│   └── conftest.ext             # 该模块的 fixtures（如有）
```

测试文件与源码模块一一对应，命名 `test_` 前缀。

## 十二、前端通用规范

以下原则跨前端框架适用（React / Vue / Svelte / 小程序均适用）。

### 12.1 状态管理

派生状态使用计算属性，禁止在渲染函数中重建对象：

```js
// 错误：每次渲染创建新对象，展开状态丢失
const fields = rawFields.map(f => ({ ...f, expanded: false }))

// 正确：用 computed 保持引用稳定
const expandedIds = ref(new Set())
const fields = computed(() =>
  rawFields.value.map(f => ({ ...f, expanded: expandedIds.value.has(f.id) }))
)
```

同样适用于 React 的 `useMemo`，Vue 的 `computed`，Svelte 的 `$derived`。

### 12.2 异步请求竞态保护

多个异步请求可能以任意顺序返回，必须保护：

```js
// 错误：旧请求可能后返回，覆盖新数据
const load = async (id) => {
  data.value = await fetchData(id)
}

// 正确：用 token/AbortController 忽略过期响应
let requestToken = 0
const load = async (id) => {
  const token = ++requestToken
  const result = await fetchData(id)
  if (token === requestToken) { data.value = result }
}
```

### 12.3 请求防重

加载中状态必须阻止重复请求：

```js
const loading = ref(false)
const load = async () => {
  if (loading.value) return    // 已在加载中，不重复发
  loading.value = true
  try { ... } finally { loading.value = false }
}
```

### 12.4 格式化函数复用

同一项目的格式化逻辑（状态文案、标签颜色、日期格式）必须抽取为公共工具函数，禁止在多个页面中重复定义。

### 12.5 样式原则

优先使用 UI 框架内置组件和变体，不写自定义 CSS。只有在框架组件无法满足的业务可视化场景（如图表），才允许有限的 inline style。

## 十三、代码格式基础

以下规则适用于所有语言，具体实现按语言惯例调整：

- **缩进**：统一缩进风格，不混用 Tab 和空格
- **导入顺序**：标准库 → 第三方库 → 本地模块，同类内按字母序
- **命名**：遵循语言惯例（如 Python snake_case、Java camelCase），项目内保持一致
- **行宽**：不设死限制，但超长行应有理由
- **文件命名**：小写字母 + 下划线或连字符（按语言惯例），禁止空格和大写混用

---

## 十四、命名规范

- **文件**：小写字母 + 下划线（`user_service.py`）或驼峰（`UserService.java`），按语言惯例
- **变量**：遵循语言惯例，常量用大写 + 下划线（`MAX_SIZE`）
- **函数**：动词开头，表达清晰动作（`getUserInfo()` / `createOrder()`）
- **类**：大驼峰命名（`UserService` / `OrderController`）

---

## 十五、注释规范

- 文件注释：功能描述、作者、创建日期
- 函数注释：参数类型、返回类型、可能异常
- 行内注释：只在非显然逻辑处加，解释"为什么"而不是"做什么"
- 禁止大段注释代码 → 直接删除，用 Git 管理历史

---

## 十六、Git 提交规范

### Commit Message 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型

- `feat`: 新功能
- `fix`: 修复 bug
- `docs`: 文档更新
- `style`: 代码格式（不影响功能）
- `refactor`: 重构
- `test`: 测试
- `chore`: 构建/工具

---

## 禁止事项

- 禁止硬编码凭证（API Key、密码）
- 禁止大段注释代码（删除或 Git 管理）
- 禁止随意 print 代替日志
- 禁止保留废弃代码、墓碑注释、兼容垃圾
- 禁止凭训练数据经验写代码，先找项目范式
- 禁止修改子类而不读父类/基类/接口
- 禁止创建无业务价值的抽象层
- 禁止为小修复创建开发计划文档
- 禁止代码上线后设计文档与代码不一致
- 禁止在审查时只跑纯函数测试，不验证真实链路
